# Add browser-based local file, folder, and archive upload support for documentation libraries

## Summary

Add support for creating ScrapeGoat documentation/vector libraries directly from files, folders, and archives uploaded through the WebUI.

ScrapeGoat currently supports local ingestion through `file://` paths, but this requires server/container filesystem access and is not usable by normal WebUI users. This issue adds a browser-based local import workflow that lets users upload and arrange documentation inside the WebUI before submitting it to the existing scraping/ingestion/vectorization engine.

A critical requirement is that the file/folder names and hierarchy shown in the WebUI upload window become the canonical source layout submitted to the ingestion engine.

This matters because ScrapeGoat includes each chunk's source location in retrieval/search reports.

Example reported source today:

```text
Source: file:///import/AC/Items-index-00931.md
```

The new upload flow must preserve this behavior by generating stable source metadata based on the user's final arranged import tree, not based on temporary upload paths.

## Background / Context

ScrapeGoat is a fork of `arabold/docs-mcp-server`.

ScrapeGoat differs from upstream in several important ways:

- SQLite storage has been replaced with PostgreSQL / Vector DB storage.
- Embedding providers, models, endpoints, and parameters are configurable.
- ScrapeGoat is not locked to mandatory external API keys.
- ScrapeGoat includes additional bugfixes and ingestion improvements.

However, the upstream ingestion architecture, supported formats, WebUI conventions, and job/pipeline design should still be used as architectural references where applicable.

Currently, local documentation ingestion requires a workflow similar to:

1. Admin creates or chooses a host directory.
2. Admin mounts that directory into the ScrapeGoat Docker container.
3. Admin copies documentation files into that mounted path.
4. User enters a `file://` path in the WebUI.
5. ScrapeGoat ingests the files into a vector library.
6. Retrieved chunks/report output includes the `file://` source path.

This workflow has several problems:

- Requires Docker/container knowledge.
- Requires server filesystem access.
- Cannot be used by normal WebUI users.
- Is unsuitable for shared/self-hosted environments.
- Makes local document ingestion effectively admin-only.
- Requires the user/admin to understand mounted container paths.
- Does not provide a convenient way to rename/restructure uploaded documentation before ingestion.

The goal of this feature is to allow users to upload and arrange documentation directly from their browser while still producing clean, meaningful source paths in retrieval results.

## Current Behavior

When clicking **Add New Documentation**, the current WebUI immediately expands the existing ingestion form.

The current form supports sources such as:

- Remote URLs
- GitHub/package-style sources, if enabled by the existing implementation
- Local filesystem paths via `file://`

There is currently no WebUI support for:

- Uploading local files from the user's device
- Uploading folders from the user's device
- Uploading archives from the user's device
- Editing the pending source tree before ingestion
- Renaming files/folders before ingestion
- Moving files/folders into a desired import hierarchy before ingestion
- Creating a documentation library from local files without server/container access
- Generating clean user-controlled source paths from uploaded content

## Expected Behavior

When the user clicks **Add New Documentation**, the WebUI should first present a source-selection flow instead of immediately expanding the existing URL ingestion form.

A centered modal should appear with the title:

```text
Documentation Source Selection
```

The modal should temporarily disable interaction with the rest of the page and present two source options:

1. **Remote URL**
2. **Local Documentation**

## Source Option: Remote URL

The **Remote URL** option should be used for websites and remote documents accessible via HTTP/HTTPS or any existing remote source type already supported by ScrapeGoat.

Selecting this option should:

- Close the source-selection modal.
- Open the existing remote ingestion panel.
- Preserve the existing remote ingestion behavior.
- Rename the panel title to:

```text
Add Remote Documentation Source
```

The remote URL/source flow must not regress.

## Source Option: Local Documentation

The **Local Documentation** option should be used for uploading documents from the user's current device.

Selecting this option should:

- Close the source-selection modal.
- Open a new local upload/import panel titled:

```text
Add Local Documentation Source
```

The local upload/import flow should allow users to create documentation/vector libraries from:

- Individual files
- Multiple files
- Entire folders
- Nested folders
- Supported archives
- Mixed file/folder/archive selections

The uploaded content must be displayed in an editable import tree before submission.

The final import tree shown in the WebUI is the source tree that must be submitted to the scraping/ingestion engine.

## Critical Requirement: WebUI Import Tree Defines Source Paths

The local upload UI is not just a visual preview.

The names and hierarchy shown in the local upload/import window must define the canonical source paths passed into the ingestion engine.

Users must be able to:

- Rename pending files before submission.
- Rename pending folders before submission.
- Create virtual folders before submission.
- Move files/folders within the pending tree before submission.
- Delete files/folders from the pending tree before submission.
- Upload folders and preserve/adjust their relative structure.
- Upload archives and view/adjust extracted contents before submission.

At submit time, the backend must normalize the final WebUI tree into the import source layout.

The ingestion engine must receive files using the final arranged tree structure, not the original client-side filenames, archive paths, temporary upload paths, or server staging paths.

## Source Metadata Requirement

Retrieved chunks/search reports currently include source metadata such as:

```text
Source: file:///import/AC/Items-index-00931.md
```

The local import feature must preserve useful source metadata while ensuring paths are derived from the final WebUI tree.

### Preferred user-facing source display

The preferred human-readable source display format is:

```text
<library> <version> > <path>
```

Example:

```text
AC Docs 1.4 > AC/Items-index-00931.md
```

If that display format is simple to implement without breaking existing source handling, use it in retrieval/report output.

### Canonical internal source URI

If ScrapeGoat's existing ingestion/reporting pipeline expects URI-like source values, use this canonical internal source URI format:

```text
file:///import/<library>/<version>/<path>
```

Example:

```text
file:///import/AC%20Docs/1.4/AC/Items-index-00931.md
```

The implementation may store both:

```text
source_uri=file:///import/<library>/<version>/<path>
source_display=<library> <version> > <path>
```

Where practical, prefer showing the friendlier `source_display` value to users while keeping `source_uri` machine-stable internally.

### Source metadata must not leak staging paths

The source metadata must not expose:

- Temporary upload directories
- Random session IDs
- Container-only staging paths
- Host bind-mount paths
- Internal cache paths
- Archive extraction scratch paths

Bad examples:

```text
Source: file:///tmp/scrapegoat-upload-817263/file.md
Source: file:///var/lib/scrapegoat/staging/session-abc123/file.md
Source: file:///srv/docker/scrapegoat/uploads/file.md
Source: file:///app/.cache/imports/tmp/uuid/file.md
```

Good examples:

```text
Source: AC Docs 1.4 > AC/Items-index-00931.md
Source: file:///import/AC%20Docs/1.4/AC/Items-index-00931.md
Source: Product Docs 2026-05-29 > API/Auth.md
```

## Local Import UI Requirements

The local import panel should provide a clear editable pending-import tree before ingestion begins.

At minimum, the UI must include:

- Library metadata fields
- File/folder/archive selection controls
- Editable pending import tree
- Rename controls
- Move/reorganize controls
- Removal of pending items before submission
- Validation status
- Upload/extraction/ingestion progress
- Clear error messages
- Final submit button

Required metadata fields:

- Library Name
- Version

Required controls:

- Add File
- Add Folder
- Add Virtual Folder
- Rename Selected
- Delete Selected
- Accept & Submit
- Cancel

The pending import tree should show:

- Files
- Folders
- Virtual folders
- Extracted archive contents
- Nested paths
- Unsupported/rejected files
- Validation errors
- Currently selected item

Tree behavior should support:

- Expand/collapse folders
- Scrollable file tree
- Highlight currently selected item
- Single click to select
- Double click to expand/collapse folder, where appropriate
- Drag/drop move within the tree
- Rename files/folders before submission
- Create folders in the pending tree
- Delete files/folders before submission

Unlike a purely visual drag/drop feature, moving files/folders in this tree must affect the submitted ingestion source layout and resulting chunk source metadata.

## Local File Upload Support

Users must be able to select and upload one or more files from their local device.

The uploader must accept files based on ScrapeGoat parser support, not file extension alone.

Validation must occur both client-side and server-side.

Client-side validation is for UX only.

Server-side parser-based validation is authoritative.

## Folder Upload Support

Users must be able to select an entire folder from their browser.

The implementation may use browser APIs such as:

```text
webkitdirectory
```

Folder upload support must include:

- Recursive folder selection
- Nested folder handling
- Preservation of relative folder paths in the pending import tree
- Ability to rename/reorganize imported folder contents before submission
- Mixed folder/file/archive upload sets

The final WebUI tree layout after user edits must be the layout submitted to the backend ingestion pipeline.

## Archive Upload Support

The uploader must support common archive formats already supported by the ingestion pipeline.

Initial required archive formats:

- `.zip`
- `.tar`
- `.tar.gz`
- `.tgz`

Archive handling must:

- Extract archives server-side.
- Preserve archive folder structure in the pending import tree.
- Allow users to rename/reorganize extracted archive contents before submission.
- Process valid files inside archives individually.
- Skip unsupported files and report them in `Scrapegoat-FailedToUpload.txt`.
- Reject nested archive entries and report them in `Scrapegoat-FailedToUpload.txt`.
- Handle nested folders correctly.
- Provide useful validation errors for invalid archives.

Archive uploads must support:

- Archives containing only supported documents
- Archives containing mixed supported/unsupported files
- Archives containing nested folders
- Multiple archives in one upload set
- Empty archives
- Corrupted archives
- Oversized archives
- Invalid archive extensions
- Archive path traversal attempts
- Nested archive entries

Nested archive entries should be rejected/skipped and reported, not recursively extracted.

The parent archive does not need to fail if it also contains valid supported files.

## Supported File Detection

Do not use file extensions as the authoritative inclusion/exclusion mechanism.

The backend should use ScrapeGoat's existing supported parser system to determine whether a file is ingestible.

Examples:

```text
BONGOS.BUM
```

If parser detection determines that this is Markdown or otherwise supported content, accept it.

```text
VIRUS.txt
```

If parser detection determines that this is binary or unsupported content, reject it even though the extension is `.txt`.

File extensions may still be used for:

- UI hints
- File picker hints
- User-facing messages
- Early warnings

But extensions must not be the authoritative security or compatibility check.

The authoritative check is:

```text
Can ScrapeGoat parse this file with a supported parser?
```

## Docker Deployment Requirement

ScrapeGoat is primarily deployed inside Docker containers.

This feature must be designed for Docker-first operation.

The implementation must not assume:

- Direct host filesystem access
- Stable host paths inside the application
- Writable container root filesystem
- Unlimited container memory
- Unlimited container writable-layer disk space
- That `/tmp` is large enough
- That WebUI/API and worker processes share a filesystem unless configured to do so

The implementation must include configurable WebUI import staging behavior suitable for Docker deployments.

# **Docker-Based Testing Requirements - REQUIRED!**

Because ScrapeGoat is primarily deployed and operated through Docker, this feature must be built and tested using Docker.

The implementation must not rely on a local host development environment that bypasses Docker-specific behavior such as container networking, mounted volumes, temporary filesystem behavior, container permissions, or PostgreSQL connectivity.

### Required Docker Test Environment

The implementer must create a temporary Docker-based test environment that includes:

- ScrapeGoat application container/image built from the current branch
- A PostgreSQL container for testing
- Any required Vector DB/container dependencies used by the current ScrapeGoat stack
- Any required worker/API/WebUI containers if the project runs them separately

The PostgreSQL database must be deployed specifically for the test run and must not reuse a developer's existing database unless explicitly configured for that purpose.

### Container and Port Safety

Before starting test containers, the implementation/test script must check existing Docker containers, networks, volumes, and host ports to avoid collisions.

Testing must use unique/randomized names for temporary resources.

Required randomized resources include:

- Container names
- Docker network names
- Docker volume names, where applicable
- Database name
- Database username, where practical
- Database password
- Host ports

Host ports must be selected from currently unused ports.

The test process must not assume that common ports are free.

Avoid hardcoded test ports such as:

```text
3000
5432
8080
````

unless the script first verifies that the port is unused.

Suggested naming pattern:

```text
scrapegoat-test-<random-string>
scrapegoat-postgres-test-<random-string>
scrapegoat-network-test-<random-string>
scrapegoat-db-test-<random-string>
```

### PostgreSQL Test Setup

The Docker test environment must start a PostgreSQL container with randomized credentials.

The implementation must configure ScrapeGoat to use that PostgreSQL instance for the test run.

The test setup must verify that:

* PostgreSQL container starts successfully
* ScrapeGoat can connect to PostgreSQL
* Required migrations/schema initialization complete successfully
* The WebUI/API can create and read documentation library state
* The local import feature successfully creates a library visible after refresh

### Docker Staging Mode Tests

The Docker test environment must test the WebUI import feature in both staging modes where implemented:

1. Memory-backed staging
2. Filesystem-backed staging

For filesystem-backed staging, the test must use a Docker volume or temporary bind mount and verify:

* The configured internal staging path is writable
* Upload/import succeeds
* Temporary files are removed after success
* Temporary files are removed after failure
* Temporary files are removed after cancellation, where cancellation is implemented

For memory-backed staging, the test must verify:

* Upload/import succeeds within configured limits
* Oversized files/archives are rejected cleanly
* The container does not crash or OOM under expected validation cases

### Required Functional Tests

The Docker-based test run must verify at least:

* Remote URL ingestion still works
* Single local file import works
* Multiple local file import works
* Folder upload/import works
* Archive upload/import works
* Unsupported files are skipped/reported correctly
* Duplicate paths are auto-renamed and reported correctly

### Cleanup Requirements

The implementer must clean up all temporary Docker resources after testing.

Cleanup must include:

* Test containers
* Test networks
* Test volumes
* Temporary bind-mount directories
* Temporary generated files
* Temporary PostgreSQL data
* Temporary `.env` or config files containing randomized test credentials

Cleanup should run even if tests fail.

The implementation should use trap/finally-style cleanup where possible.


* Which staging modes were tested

## WebUI Import Staging Storage Modes

WebUI-imported libraries must support configurable temporary staging while ingestion is in progress.

The implementation should provide at least two staging modes:

1. RAM / memory-backed staging
2. Local directory / filesystem-backed staging

Suggested config:

```text
SCRAPEGOAT_WEBUI_IMPORT_STAGING_MODE=memory
SCRAPEGOAT_WEBUI_IMPORT_STAGING_MODE=filesystem
```

### Default staging mode

Default staging mode:

```text
memory
```

Suggested config:

```text
SCRAPEGOAT_WEBUI_IMPORT_STAGING_MODE=memory
```

Memory mode must still enforce upload, archive, extracted-size, and file-count limits.

Since ScrapeGoat runs inside Docker, memory staging must be bounded so large uploads or decompression bombs do not OOM-kill the container.

### Mode: memory

When using memory-backed staging:

- Uploaded files are stored in memory or memory-backed temporary storage while the import is in progress.
- Files must be discarded after ingestion, failure, cancellation, or manual cleanup.
- Memory limits must be strictly enforced.
- Archive extraction limits must be strictly enforced.
- This mode is convenient for small imports but dangerous for large archives if limits are weak.

Docker-specific note:

Memory-backed staging may count against the container memory limit and can cause container OOM kills if not carefully bounded.

If implemented using Docker `tmpfs`, the documentation should show how to configure a safe tmpfs size.

Example Docker Compose concept:

```yaml
services:
  scrapegoat:
    tmpfs:
      - /run/scrapegoat-imports:size=1g,mode=1770
```

Suggested memory staging path:

```text
SCRAPEGOAT_WEBUI_IMPORT_MEMORY_PATH=/run/scrapegoat-imports
```

### Mode: filesystem

When using filesystem-backed staging:

- Uploaded files are staged in a local directory inside the container.
- The directory may be backed by a Docker volume or bind mount.
- Files remain temporary processing inputs, not permanent documentation libraries.
- Cleanup must still occur after success, failure, or cancellation.
- Failed import sessions should not be retained for debugging; log failures normally.

Default internal filesystem staging path pattern:

```text
/tmp/scrapegoat-import-<random-uuid>
```

Suggested config:

```text
SCRAPEGOAT_WEBUI_IMPORT_STAGING_MODE=filesystem
SCRAPEGOAT_WEBUI_IMPORT_STAGING_INTERNAL_PATH=/tmp
```

Each upload/import session should create a unique directory such as:

```text
/tmp/scrapegoat-import-2f9d7b6e-5df1-4fb2-b7d2-7dfc7ac5f7db
```

Optional external host path setting for Docker documentation/diagnostics:

```text
SCRAPEGOAT_WEBUI_IMPORT_STAGING_EXTERNAL_PATH=/srv/scrapegoat/import-staging
```

Important:

`SCRAPEGOAT_WEBUI_IMPORT_STAGING_INTERNAL_PATH` is the path the container uses.

`SCRAPEGOAT_WEBUI_IMPORT_STAGING_EXTERNAL_PATH` is the optional host-side path used for documentation, diagnostics, and Docker Compose generation/help text. The application must not assume the host path exists inside the container unless it is bind-mounted.

Example Docker Compose concept:

```yaml
services:
  scrapegoat:
    volumes:
      - /srv/scrapegoat/import-staging:/tmp
    environment:
      SCRAPEGOAT_WEBUI_IMPORT_STAGING_MODE: filesystem
      SCRAPEGOAT_WEBUI_IMPORT_STAGING_INTERNAL_PATH: /tmp
      SCRAPEGOAT_WEBUI_IMPORT_STAGING_EXTERNAL_PATH: /srv/scrapegoat/import-staging
```

The filesystem staging path must not leak into chunk source metadata.

Even if files are temporarily staged at:

```text
/tmp/scrapegoat-import-2f9d7b6e-5df1-4fb2-b7d2-7dfc7ac5f7db/root/AC/Items-index-00931.md
```

the retrieved source should still be:

```text
Source: AC Docs 1.4 > AC/Items-index-00931.md
```

or:

```text
Source: file:///import/AC%20Docs/1.4/AC/Items-index-00931.md
```

## Upload Storage Model

Uploaded files must be treated as ephemeral processing inputs only.

Uploaded files must not become permanent filesystem libraries.

The implementation may use:

- Streamed multipart upload handling
- Memory-backed staging
- Docker tmpfs staging
- Temporary filesystem staging
- Docker volume-backed staging
- A hybrid approach

However, the required behavior is:

- Uploads exist only long enough for validation, extraction, ingestion, vectorization, and verification.
- Uploaded files are deleted after successful ingestion.
- Extracted archive contents are deleted after successful ingestion.
- Failed ingestion attempts clean up temporary files.
- Cancelled uploads clean up temporary files.
- Failed import sessions are not retained for debugging; failures are logged normally.
- Long-term document persistence outside the vector DB is not desired.

Do not implement permanent paths such as:

```text
/uploads/<library-name>/
```

as long-lived user documentation stores.

Temporary processing directories are acceptable if they are cleaned up.

## Upload Session Expiration

Upload session expiration should be treated as infinite while the process/session is active.

Suggested behavior:

```text
SCRAPEGOAT_WEBUI_IMPORT_SESSION_TTL_SECONDS=0
```

Where `0` means no time-based expiry.

However, the implementation should still clean up sessions on:

- Explicit cancellation
- Failed upload
- Failed ingestion
- Successful ingestion
- Application/container startup if stale temporary sessions are found

Docker gotcha:

Even with infinite TTL, stale `/tmp/scrapegoat-import-*` directories may remain after a hard container crash/restart. Startup cleanup should remove stale import directories that are not associated with an active job.

## Canonical Import Root

The implementation should define a canonical virtual import root used for source metadata when URI-style metadata is required.

Suggested default:

```text
file:///import
```

Suggested config:

```text
SCRAPEGOAT_WEBUI_IMPORT_SOURCE_URI_ROOT=file:///import
```

The final source path should be built from:

```text
<source-uri-root>/<library>/<version>/<final-webui-tree-relative-path>
```

Example:

```text
file:///import/AC%20Docs/1.4/AC/Items-index-00931.md
```

The implementation should also support the human-readable display form where practical:

```text
<library> <version> > <final-webui-tree-relative-path>
```

Example:

```text
AC Docs 1.4 > AC/Items-index-00931.md
```

Library name, version, and path components must be sanitized and stable.

The implementation must document the chosen storage/display behavior.

## Path Normalization Rules

Because the WebUI tree defines source metadata, path normalization must be explicit and safe.

The implementation must define how it handles:

- Leading slashes
- Backslashes from Windows clients
- `.` path segments
- `..` path segments
- Duplicate slashes
- Empty path components
- Unicode normalization
- Control characters
- Newlines in filenames
- Very long filenames
- Very long paths
- Case-sensitive conflicts
- Reserved Windows-style names, if relevant
- Duplicate filenames in the same folder

User-provided names must never allow escaping the canonical import root.

Invalid names should be rejected with a clear error.

The final source paths should be deterministic.

## Validation Rules

Validation must happen server-side before ingestion.

Validation must include:

- Parser-based support detection
- File content detection where practical
- Size limit validation
- Archive format validation
- Archive extraction safety checks
- File count validation
- Folder depth validation
- Final source path validation
- Empty upload set validation
- Duplicate library/version validation, if applicable

File extensions must not be used as the authoritative inclusion/exclusion mechanism.

Unsupported files should not silently disappear.

Unsupported files inside uploaded folders or archives should not cause the entire import to fail.

Behavior:

1. Import all supported files.
2. Skip unsupported files.
3. Show a clear message to the user that some files were not imported.
4. Generate a downloadable report named:

```text
Scrapegoat-FailedToUpload.txt
```

The report should list every file that was not imported and the reason when available.

Example report content:

```text
ScrapeGoat failed to import the following files:

Unsupported or unreadable:
- AC/raw/VIRUS.txt
  Reason: file appears to be binary and no supported parser accepted it

- AC/archive/old-format.xyz
  Reason: no supported parser accepted this file

Rejected:
- AC/nested/docs.zip
  Reason: nested archives are not supported
```

After processing, the browser should prompt the user to download the report.

The user can accept or decline the download in their browser.

Recommended behavior:

- Directly selected unsupported files: reject/skip with clear report entry.
- Unsupported files inside archives/folders: skip with warning/report entry.
- Empty archives: reject with clear error/report entry.
- Archives containing no supported files: reject with clear error/report entry.
- Mixed valid/invalid archive contents: ingest valid files and report skipped files.
- Nested archives: reject/skip nested archive entry and report it.

## Duplicate Handling

The implementation must define duplicate behavior for:

- Duplicate filenames
- Duplicate relative paths
- Duplicate extracted archive paths
- Duplicate files created by user tree edits
- Duplicate archives
- Duplicate library names
- Duplicate library name + version combinations

### Duplicate file/path behavior

Duplicate final paths should be auto-renamed.

The implementation should use deterministic, readable renaming.

Example:

```text
Guide.md
Guide (1).md
Guide (2).md
```

or:

```text
Guide.md
Guide-1.md
Guide-2.md
```

The chosen format should be consistent.

When auto-renames occur, the system should generate a downloadable report named:

```text
Scrapegoat-RenamedFiles.txt
```

Example report content:

```text
ScrapeGoat automatically renamed files to avoid duplicate source paths:

- Original: AC/Guide.md
  Renamed:  AC/Guide (1).md

- Original: AC/API/Auth.md
  Renamed:  AC/API/Auth (1).md
```

The final renamed paths are the paths submitted to the ingestion engine and used for source metadata.

### Duplicate library/version behavior

Duplicate library name/version combinations must be rejected before ingestion starts.

Behavior:

1. Detect the duplicate before uploading/ingesting where possible.
2. Inform the user that the library/version already exists.
3. Show the latest existing version.
4. Suggest incrementing the version by 1 or using a date-based version.
5. Allow the user to create a new version.

Do not overwrite an existing library/version during this flow.

Example message:

```text
A library named "AC Docs" with version "1.4" already exists.

Latest existing version: 1.4

Please choose a new version, for example:
- 1.5
- 2026-05-29
```

## Security Requirements

The implementation must explicitly protect against unsafe upload/archive behavior.

### Archive Security

Prevent:

- Zip-slip/path traversal attacks
- Absolute path extraction
- `../` path traversal
- Symlink extraction abuse
- Hardlink extraction abuse
- Device/special file extraction
- Archive bombs
- Decompression bombs
- Excessive nested directory depth
- Excessive file count
- Recursive archive abuse

Archive extraction must never write outside the intended temporary extraction directory.

Archive paths must be normalized before they are shown in the WebUI tree.

The user may reorganize safe extracted files after extraction, but unsafe archive entries must never be allowed into the pending tree.

### Upload Security

Prevent:

- Unsupported file injection
- Oversized uploads
- Invalid content spoofing where practical
- Malformed archive crashes
- Arbitrary server file overwrite
- Executable persistence locations
- Path collision attacks
- Filename/path control character issues
- Source path metadata spoofing

### Filesystem Safety

Uploaded or extracted content must never:

- Escape temporary storage areas
- Overwrite arbitrary server files
- Create permanent executable files
- Become a long-term source filesystem library
- Be served publicly by the WebUI unless explicitly intended and safe

## Docker-Specific Gotchas To Account For

The implementation and documentation should account for Docker-specific behavior.

### Container writable layer

Do not rely on the container writable layer for large uploads.

Large uploads/extractions can fill Docker overlay storage and potentially affect the host.

Prefer:

- Explicit tmpfs for memory mode
- Explicit Docker volume or bind mount for filesystem mode

### Bind mount permissions

If filesystem staging uses a bind mount, the host path must be writable by the container user.

The documentation should mention:

- Container UID/GID
- Host directory ownership
- Host directory permissions
- Rootless Docker considerations

### SELinux/AppArmor

On SELinux-enabled hosts, bind mounts may need labels such as `:z` or `:Z`.

Example:

```yaml
volumes:
  - /srv/scrapegoat/import-staging:/tmp:Z
```

The documentation should avoid assuming all Docker hosts behave like Ubuntu/Debian.

### Read-only root filesystem

Some deployments may run containers with a read-only root filesystem.

The implementation should allow staging paths to be configured to writable mounted locations.

### Reverse proxy upload limits

Many Docker deployments sit behind nginx, Caddy, Traefik, Cloudflare Tunnel, or another reverse proxy.

The documentation should remind users that large uploads may also require reverse proxy config changes.

Examples:

- nginx `client_max_body_size`
- nginx proxy read/send timeouts
- Traefik body size / buffering middleware
- Caddy request body limits
- Cloudflare upload limits, where applicable

### Multi-container deployments

Even though deployment is Docker-based, the project may still run as multiple containers, for example:

- WebUI/API container
- Worker container
- Database container
- Vector DB container

If upload handling and ingestion run in different containers, staging storage must be shared or the payload must be transferred to the worker.

The implementation must not assume that a path created inside the WebUI/API container exists inside the worker container.

### Container restarts

If the container restarts during upload or ingestion:

- In-progress sessions should be marked failed or cleaned up.
- Partial staging files should be cleaned up at next startup.
- The UI should not show stale upload sessions as active forever.

### Memory limits

Memory-backed staging must respect container memory limits.

A compressed archive can expand far beyond its compressed size.

The implementation must enforce:

- Compressed archive limit
- Extracted archive limit
- File count limit
- Total upload limit

### Disk limits

Filesystem-backed staging must handle out-of-space errors cleanly.

If the staging volume fills up, the user should receive a clear error and partial files should be cleaned up.

### Source path leakage

Chunk source metadata must not reveal host paths, Docker volume paths, or temporary container paths.

Do not expose paths such as:

```text
/srv/scrapegoat/import-staging
/var/lib/docker/volumes/...
/tmp/scrapegoat-import-...
/var/lib/scrapegoat/import-staging/session-...
```

Retrieval source metadata should use the canonical virtual source URI and/or display source derived from the final WebUI import tree.

## Confirmed Implementation Decisions

### Canonical source URI / display format

Preferred user-facing display:

```text
<library> <version> > <path>
```

Fallback/internal URI format:

```text
file:///import/<library>/<version>/<path>
```

Best implementation if practical:

```text
source_uri=file:///import/<library>/<version>/<path>
source_display=<library> <version> > <path>
```

### Default upload limits

```text
Single file size:           100MB
Total upload size:          999MB
Compressed archive size:    500MB
Extracted archive size:     999MB
File count:                 999
Folder/archive depth:       9
Filename length:            99
Final source path length:   255
```

### Unsupported files

Import supported files, skip unsupported files, show warning, and generate:

```text
Scrapegoat-FailedToUpload.txt
```

### Duplicate filenames/paths

Auto-rename duplicates deterministically and generate:

```text
Scrapegoat-RenamedFiles.txt
```

### Duplicate library/version

Reject before ingestion, show latest version, and suggest incrementing the version or using a date.

### Default staging mode

```text
memory
```

### Default filesystem staging path

```text
/tmp/scrapegoat-import-<random-uuid>
```

### Failed import session retention

Do not retain failed sessions for debugging. Log normally and clean up.

### Upload cancellation

Cancelled upload/import means no library is created.

### Upload session expiration

No time-based expiry while active.

Use:

```text
SCRAPEGOAT_WEBUI_IMPORT_SESSION_TTL_SECONDS=0
```

where `0` means no time-based expiry.

Still clean up after success, failure, cancellation, and stale startup state.

### Nested archives

Reject/skip nested archives and report them in:

```text
Scrapegoat-FailedToUpload.txt
```

### Supported file detection

Do not use file extensions as the authoritative exclusion mechanism.

Use ScrapeGoat's parser support to decide whether a file can be imported.

### Successful Vector DB verification

Successful ingestion means:

```text
Job completed + document management service confirms library exists + library visible after refresh
```

## Implementation Notes

Prefer a first implementation that reliably supports:

- Source-selection modal
- Remote flow preservation
- File upload
- Folder upload
- Archive upload
- Editable pending import tree
- Rename/move/delete before submission
- Final source path preview
- Parser-based backend validation
- Secure archive extraction
- Duplicate auto-renaming
- Failed upload report generation
- Renamed file report generation
- Docker-aware staging configuration
- Memory default staging
- Optional filesystem staging
- Existing pipeline integration
- Progress/status feedback
- Cleanup

The most important outcome is that a normal WebUI user can create a documentation/vector library from local files without needing Docker, server filesystem access, mounted paths, or `file://` knowledge, while still producing clean and meaningful retrieval source paths based on the final arranged WebUI import tree.
